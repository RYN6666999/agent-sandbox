# Meta 協議：撰寫協議提示詞

> **用途**：寫一份新的協議模板時的指引。
> **適用對象**：Scream 與所有需要定義新協議的 agent。

---

## 啟動條件

當需要定義新的 agent 之間溝通方式時：

- [x] 引入新類型的 agent（如：新的 executor、新的審查者）
- [x] 現有協議不涵蓋的新場景
- [x] 發現現有協議有漏洞（格式不明確、缺少錯誤處理）

---

## 協議模板結構

每一份協議應包含以下區塊：

```markdown
# 協議：<協議名稱>

> **用途**：<一句話說明>
> **通訊媒介**：<gbrain / executor registry / knowledge API / 直接輸出>
> **延遲特性**：<同步/非同步，秒/分鐘/小時級>
```

## 撰寫原則

1. **一個協議只做一件事**
   - 交接、紀錄、審查、拆解要分開
   - 不要為了省檔案把多個場景混在一起

2. **格式要夠具體**
   - 不要寫「提供上下文」
   - 要寫「提供以下欄位：key, content, metadata（含 type/status/agent）」

3. **包含錯誤路徑**
   - 對方不回怎麼辦
   - 格式不對怎麼辦
   - 對方拒答怎麼辦
   - 寫入失敗怎麼辦

4. **寫驗收標準**
   - 協議執行完應該有明確的完成狀態
   - 成功、失敗、需升級 — 三者都要定義

5. **從實際案例提煉**
   - 先發生過問題 → 再寫協議
   - 不要預先寫太複雜的協議

---

## 範例：撰寫一份「程式碼審查請求」協議

依循上述結構，Scream 應該先確認：

**啟動條件**：什麼時候需要程式碼審查？
- 所有 PR merge 前？
- 只有改到核心模組時？
- 只有 Scream 不確定時？

**訊息格式**：審查請求要包含什麼？
- diff 內容？
- 為什麼這樣改？
- 測試結果？

**回覆格式**：審查者怎麼回？
- Approved / Changes requested / Blocked
- 要不要附修改建議？

**錯誤處理**：審查者超過 24h 沒回？
- 自動 merge？發 reminder？升級？

先把這些問題想清楚再寫，協議就會紮實。
